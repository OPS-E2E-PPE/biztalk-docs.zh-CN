## <a name="the-role-of-the-pipeline"></a><span data-ttu-id="b4faf-119">管道的作用</span><span class="sxs-lookup"><span data-stu-id="b4faf-119">The Role of the Pipeline</span></span>  
 <span data-ttu-id="b4faf-120">尽管初始消息是由适配器创建的，但对收到消息的处理过程大部分发生在接收管道中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-120">While it is the adapter that creates the initial message, most of the processing that occurs on a received message happens in the receive pipeline.</span></span> <span data-ttu-id="b4faf-121">管道处理针对消息内容以及消息上下文。</span><span class="sxs-lookup"><span data-stu-id="b4faf-121">Pipeline processing deals with message content as well as message context.</span></span> <span data-ttu-id="b4faf-122">消息内容通常在解码、拆装和验证阶段进行处理，而对消息上下文的处理则可发生在所有阶段。</span><span class="sxs-lookup"><span data-stu-id="b4faf-122">Message content is generally handled in the decoding, disassembling, and validating stages, while message context can be handled in all stages.</span></span> <span data-ttu-id="b4faf-123">但是，管道无需作用于内容或上下文。</span><span class="sxs-lookup"><span data-stu-id="b4faf-123">A pipeline, however, does not necessarily act on either the content or the context.</span></span> <span data-ttu-id="b4faf-124">例如，默认的直通管道并未配置任何组件并且不会对消息内容或上下文进行处理。</span><span class="sxs-lookup"><span data-stu-id="b4faf-124">For example, the default pass-through pipeline has no components configured and performs no processing on the message content or context.</span></span> <span data-ttu-id="b4faf-125">为简单起见，此文档将重点介绍拆装组件，因为这些组件通常对消息路由的影响最大。</span><span class="sxs-lookup"><span data-stu-id="b4faf-125">For simplicity, this document focuses on the disassembling components as they generally have the greatest impact on message routing.</span></span>  
  
 <span data-ttu-id="b4faf-126">*拆装器* 的工作是处理来自适配器的传入消息并将其拆装成多个消息，以及对消息数据进行解析。</span><span class="sxs-lookup"><span data-stu-id="b4faf-126">The job of the *disassembler* is to process an incoming message from an adapter and to disassemble it into many messages, and parse the message data.</span></span> <span data-ttu-id="b4faf-127">在传入消息具有多个较小的消息时，称为 *交换*。</span><span class="sxs-lookup"><span data-stu-id="b4faf-127">When an incoming message has many smaller messages, this is known as an *interchange*.</span></span> <span data-ttu-id="b4faf-128">通过使开发人员能够配置与包装内容（即，平面文件拆装器的头部架构和尾部架构以及 XML 拆装器的信封架构）以及（可能重复）正文内容有关的信息，平面文件拆装器和 XML 拆装器可以处理交换。</span><span class="sxs-lookup"><span data-stu-id="b4faf-128">Both the flat file disassembler and the XML disassembler handle interchanges by enabling a developer to configure information about the wrapping content (that is, a header and trailing schema for the flat file disassembler and an envelope schema for the XML disassembler) and the (potentially repeating) body content.</span></span> <span data-ttu-id="b4faf-129">此外，这两种拆装器都可将原始消息解析为 XML 内容。</span><span class="sxs-lookup"><span data-stu-id="b4faf-129">In addition, both of these disassemblers parse the original message into XML content.</span></span> <span data-ttu-id="b4faf-130">如果在 BizTalk Server 中不需要做进一步的 XML 处理，则自定义拆装器无需将内容解析为 XML。</span><span class="sxs-lookup"><span data-stu-id="b4faf-130">A custom disassembler does not necessarily parse the content into XML if further XML processing in BizTalk Server is not required.</span></span> <span data-ttu-id="b4faf-131">示例方案包括简单的路由方案，在此方案中，在特定接收位置进入系统的消息将发送到不具有映射或其他基于 XML 的处理的特定发送端口。</span><span class="sxs-lookup"><span data-stu-id="b4faf-131">An example scenario might include a simple routing situation in which messages entering the system at a particular receive location are sent to a specific send port with no mapping or other XML-based processing.</span></span>

<SomeOtherElement type="sample"/>

 <span data-ttu-id="b4faf-137">若要在路由中使用消息类型，则必须将其升级到上下文中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-137">To use message type in routing, it must be promoted into the context.</span></span> <span data-ttu-id="b4faf-138">拆装器用于将此值升级到消息上下文以及带有最具体的消息结构信息的管道组件中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-138">Disassemblers are used to promote this value into the message context as well as the pipeline components with the most specific knowledge of message structure.</span></span> <span data-ttu-id="b4faf-139">XML 和平面文件拆装器将在处理消息时升级消息类型，并且任何自定义拆装器也应升级此属性以确保正确地进行路由。</span><span class="sxs-lookup"><span data-stu-id="b4faf-139">The XML and Flat File disassemblers promote the message type as they are processing messages, and any custom disassembler should also promote this property to ensure proper routing.</span></span>  
  
 <span data-ttu-id="b4faf-140">需要特别注意的是，消息不必一定具有某一类型。</span><span class="sxs-lookup"><span data-stu-id="b4faf-140">It is important to note that a message is not required to have a type.</span></span> <span data-ttu-id="b4faf-141">如前所述，消息各部分可以是任何二进制数据，并且无需具有定义其结构的架构。</span><span class="sxs-lookup"><span data-stu-id="b4faf-141">As mentioned previously, the parts of a message can be any binary data and need not have a schema that defines their structure.</span></span> <span data-ttu-id="b4faf-142">尽管自定义管道组件、适配器或从业务流程调用的代码均可与此类型的消息部分交互，但这些部分通常不必由 BizTalk Server 本身对其进行大量处理（如果有）即可在整个 BizTalk Server 中传递。</span><span class="sxs-lookup"><span data-stu-id="b4faf-142">This type of message part is generally passed through BizTalk Server without much, if any, processing done on it by BizTalk Server itself, though custom pipeline components, adapters, or code called from orchestrations may interact with these parts.</span></span>  
  
 <span data-ttu-id="b4faf-143">管道组件（例如适配器）还可将属性写入和升级到消息上下文。</span><span class="sxs-lookup"><span data-stu-id="b4faf-143">Pipeline components, like adapters, also write and promote properties into the message context.</span></span> <span data-ttu-id="b4faf-144">实际上，管道组件是大多数开发人员用于将属性放入消息上下文的最常见的机制。</span><span class="sxs-lookup"><span data-stu-id="b4faf-144">In fact, pipeline components are the most common mechanism most developers use to get properties into the message context.</span></span> <span data-ttu-id="b4faf-145">开发人员可创建架构并在架构中升级属性。</span><span class="sxs-lookup"><span data-stu-id="b4faf-145">Developers create schemas and can promote properties in the schema.</span></span> <span data-ttu-id="b4faf-146">此信息将作为批注存储于架构中，以供随后管道组件使用。</span><span class="sxs-lookup"><span data-stu-id="b4faf-146">This information is stored in the schema as annotations which can then be used by pipeline components.</span></span> <span data-ttu-id="b4faf-147">所有内置拆装器和组装器组件（平面文件、XML 和 BizTalk 框架）均使用文档架构检索与要升级的属性有关的信息。</span><span class="sxs-lookup"><span data-stu-id="b4faf-147">All of the built-in disassembler and assembler components - FlatFile, XML, and BizTalk Framework - use the document schema to retrieve information about the properties that are to be promoted.</span></span> <span data-ttu-id="b4faf-148">使用来自批注的 XML Path 语言 (XPath) 语句，拆装器可以知道要升级的元素在文档中的位置。</span><span class="sxs-lookup"><span data-stu-id="b4faf-148">Using the XML Path Language (XPath) statement from the annotations, the disassembler knows the location in the document of elements to be promoted.</span></span> <span data-ttu-id="b4faf-149">在通过文档传输过程中，拆装器将查找与 XPath 语句之一相匹配的元素，并根据需要，将值升级或写入上下文中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-149">During the process of streaming through the document, the disassembler finds those elements that match one of the XPath statements and promotes or writes the value into the context as appropriate.</span></span>

 <span data-ttu-id="b4faf-156">具有基本类型 MessageDataPropertyBase 的属性表示此属性的值来自消息的内容。</span><span class="sxs-lookup"><span data-stu-id="b4faf-156">A property with a base type of MessageDataPropertyBase indicates that the value for this property comes from the content of the message.</span></span> <span data-ttu-id="b4faf-157">这是在属性架构中定义的属性的默认值，并且最常用。</span><span class="sxs-lookup"><span data-stu-id="b4faf-157">This is the default value for properties defined in a property schema and is the most common usage.</span></span> <span data-ttu-id="b4faf-158">而 MessageContextPropertyBase 则表示属性属于消息上下文，但不必直接来自消息数据。</span><span class="sxs-lookup"><span data-stu-id="b4faf-158">MessageContextPropertyBase indicates a property that is intended to be part of the message context but does not necessarily come from the message data directly.</span></span> <span data-ttu-id="b4faf-159">以 MessageContextPropertyBase 作为其基本类型的属性通常由适配器和拆装器升级，并包含通用属性（例如，消息类型和适配器类型）。</span><span class="sxs-lookup"><span data-stu-id="b4faf-159">Properties with MessageContextPropertyBase as their base type are often promoted by adapters and disassemblers and include common properties such as message type and adapter type.</span></span>  
  
 <span data-ttu-id="b4faf-160">了解属性的各种类型并在定义属性时适当使用这些类型十分重要。</span><span class="sxs-lookup"><span data-stu-id="b4faf-160">It is important to understand the different types and use them appropriately when defining properties.</span></span> <span data-ttu-id="b4faf-161">而对于在业务流程中访问消息的上下文属性时，了解这一点尤为重要。</span><span class="sxs-lookup"><span data-stu-id="b4faf-161">One of the most significant implications occurs when accessing context properties for a message in an orchestration.</span></span> <span data-ttu-id="b4faf-162">如果属性标识为 MessageDataPropertyBase，则业务流程设计器将检查要接收消息的架构，并确保该架构定义了匹配的升级属性。</span><span class="sxs-lookup"><span data-stu-id="b4faf-162">If a property is identified as a MessageDataPropertyBase, the Orchestration Designer examines the schema of the message being received and ensures that it defines a matching promoted property.</span></span> <span data-ttu-id="b4faf-163">如果在与要访问的升级属性相关联的架构中没有找到任何属性，则设计器将不允许对其进行访问。</span><span class="sxs-lookup"><span data-stu-id="b4faf-163">If no property is found in the schema tied to the promoted property being accessed, then the Designer does not allow you to access it.</span></span> <span data-ttu-id="b4faf-164">相反，如果该属性定义为 MessageContextPropertyBase，则该消息类型将不起作用并可访问该属性。</span><span class="sxs-lookup"><span data-stu-id="b4faf-164">On the other hand, if the property is defined as being a MessageContextPropertyBase, the message type does not matter and the property can be accessed.</span></span>  
  
 <span data-ttu-id="b4faf-165">在自定义管道中，将属性升级或写入到上下文的机制非常相似。</span><span class="sxs-lookup"><span data-stu-id="b4faf-165">In custom pipelines, the mechanism for promoting or writing properties to the context is very similar.</span></span> <span data-ttu-id="b4faf-166">对于写入属性，可通过调用 IBaseMessageContext.Write 方法将值放置在上下文中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-166">For writing properties, you use a call to the IBaseMessageContext Write method to place the value in the context.</span></span> <span data-ttu-id="b4faf-167">而对于已升级属性，只需使用 IBaseMessageContext.Promote 方法。</span><span class="sxs-lookup"><span data-stu-id="b4faf-167">For promoted properties, you simply use the IBaseMessageContext Promote method instead.</span></span> <span data-ttu-id="b4faf-168">上述每种方法均采用属性名称、命名空间和值。</span><span class="sxs-lookup"><span data-stu-id="b4faf-168">Each of these methods takes a property name, a namespace, and value.</span></span> <span data-ttu-id="b4faf-169">对于升级属性，名称和命名空间为属性架构中定义的属性的名称和命名空间，并可通过引用属性架构程序集和使用为属性创建的类上的属性，更轻松地对其进行访问。</span><span class="sxs-lookup"><span data-stu-id="b4faf-169">For the promoted properties, the name and namespace are those of the property defined in the property schema and are most easily accessed by referencing the property schema assembly and using the properties on the class created for the property.</span></span> <span data-ttu-id="b4faf-170">而可分辨字段则使用公共命名空间 http://schemas.microsoft.com/BizTalk/2003/btsDistinguishedFields，并且用于检索值的 XPath 表达式通常用作名称。</span><span class="sxs-lookup"><span data-stu-id="b4faf-170">Distinguished fields use a common namespace, http://schemas.microsoft.com/BizTalk/2003/btsDistinguishedFields, and the XPath expression used to retrieve the value is usually used as the name.</span></span>  
  
 <span data-ttu-id="b4faf-171">下面的代码显示了将属性升级和写入到上下文的示例。</span><span class="sxs-lookup"><span data-stu-id="b4faf-171">The code below shows an example of both promoting and writing properties into the context.</span></span> <span data-ttu-id="b4faf-172">请注意，在此示例中，可分辨字段将写入到上下文中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-172">Note that in this example, a distinguished field is being written into the context.</span></span> <span data-ttu-id="b4faf-173">这只适用于以下类型的业务流程：在该业务流程中，消息架构将标识可分辨字段以便业务流程设计器了解该字段。</span><span class="sxs-lookup"><span data-stu-id="b4faf-173">This is only useful for orchestrations in which the message schema identifies the distinguished field so that the Orchestration Designer knows about the field.</span></span> <span data-ttu-id="b4faf-174">将属性写入到上下文中有助于接收或发送端上的其他管道组件使用这些属性。</span><span class="sxs-lookup"><span data-stu-id="b4faf-174">It may be useful to write properties into the context for use by other pipeline components on the receiving or sending side.</span></span>  
  
```  
//create an instance of the property to be promoted  
SOAP.MethodName methodName = new SOAP.MethodName();  
  
//call the promote method on the context using the property class for name   
//and namespace  
pInMsg.Context.Promote(methodName.Name.Name, methodName.Name.Namespace,   
"theSOAPMethodName");  
  
//write a distinguished field to the context  
pInMsg.Context.Write("theDistinguishedProperty",   
"http://schemas.microsoft.com/BizTalk/2003/btsDistinguishedFields",   
"theDistinguishedValue");  
```  
  
 <span data-ttu-id="b4faf-175">在将值写入或升级到上下文中时请注意以下事实：</span><span class="sxs-lookup"><span data-stu-id="b4faf-175">Keep the following facts in mind when writing or promoting values into the context:</span></span>  
  
-   <span data-ttu-id="b4faf-176">如果使用先前曾用于升级属性的同一名称和命名空间将值写入到上下文中，将导致无法再升级该属性。</span><span class="sxs-lookup"><span data-stu-id="b4faf-176">Writing a value into the context with the same name and namespace that were used previously to promote the property causes that property to no longer be promoted.</span></span> <span data-ttu-id="b4faf-177">写入的内容将从根本上覆盖升级的内容。</span><span class="sxs-lookup"><span data-stu-id="b4faf-177">The write essentially overwrites the promotion.</span></span>  
  
-   <span data-ttu-id="b4faf-178">将空值写入到上下文中将删除该值，因为不允许使用空值属性。</span><span class="sxs-lookup"><span data-stu-id="b4faf-178">Writing a value of null into the context deletes the value, because null-valued properties are not permitted.</span></span>  
  
-   <span data-ttu-id="b4faf-179">升级属性的长度限制在 256 个字符之内，而写入属性则没有长度上的限制。</span><span class="sxs-lookup"><span data-stu-id="b4faf-179">Promoted properties are limited to 256 characters in length while written properties have no length limitation.</span></span>  
  
 <span data-ttu-id="b4faf-180">升级属性用于消息路由，并由于有关比较和存储效率的原因，对升级属性的大小有所限制。</span><span class="sxs-lookup"><span data-stu-id="b4faf-180">Promoted properties are used in message routing and are limited in size for reasons of efficiency in comparison and storage.</span></span> <span data-ttu-id="b4faf-181">尽管对写入属性的大小没有硬性限制，但如果在上下文中使用过大的值，也会对性能造成影响，因为这些值仍必须与消息一起进行处理和传递。</span><span class="sxs-lookup"><span data-stu-id="b4faf-181">While written properties have no hard limits on size, using excessively large values in the context will have an impact on performance, because those values must still be processed and passed with the message.</span></span>  
  
 <span data-ttu-id="b4faf-182">在消息准备从 BizTalk Server 发送时，它将在发送端口中经历一个互补的过程。</span><span class="sxs-lookup"><span data-stu-id="b4faf-182">When a message is ready to be sent from BizTalk Server, it undergoes a complementary process in the send port.</span></span> <span data-ttu-id="b4faf-183">映射将在发送管道执行前应用于消息，从而，消息在由管道处理并通过适配器发送前转换为特定于客户或应用程序的格式。</span><span class="sxs-lookup"><span data-stu-id="b4faf-183">Maps are applied to messages before the send pipeline is executed, allowing a message to be transformed to a customer or application-specific format before being processed by the pipeline and sent through the adapter.</span></span> <span data-ttu-id="b4faf-184">在发送管道中，属性将从上下文降级到消息中，而非升级到消息上下文中。</span><span class="sxs-lookup"><span data-stu-id="b4faf-184">In the send pipeline, instead of promoting properties into the message context, properties are demoted from the context into the message.</span></span>
