## <a name="deploying-orchestrations"></a><span data-ttu-id="df89c-120">部署业务流程</span><span class="sxs-lookup"><span data-stu-id="df89c-120">Deploying Orchestrations</span></span>  
 <span data-ttu-id="df89c-121">当在设计时指定绑定时，部署业务流程时，创建与业务流程中配置的参数相匹配的物理端口。</span><span class="sxs-lookup"><span data-stu-id="df89c-121">When a binding is specified at design time, the physical port that matches the parameters configured in the orchestration is created when the orchestration is deployed.</span></span> <span data-ttu-id="df89c-122">当在部署时配置绑定时，符合要求的逻辑端口的任何端口可以绑定到业务流程端口中。</span><span class="sxs-lookup"><span data-stu-id="df89c-122">When the binding is configured at deployment time, any port that matches the requirements of the logical port can be bound to the orchestration port.</span></span> <span data-ttu-id="df89c-123">对于动态绑定，如使用立即指定选项，只需创建物理端口，但是该端口是没有配置地址信息的动态发送端口。</span><span class="sxs-lookup"><span data-stu-id="df89c-123">For dynamic binding, a physical port is created just as with the Specify Now option, but the port is a dynamic send port that has no address information configured.</span></span>  
  
 <span data-ttu-id="df89c-124">令人困惑的概念是： 尽管业务流程中的发送端口绑定到物理发送端口，但这不会阻止消息送达到其他订户。</span><span class="sxs-lookup"><span data-stu-id="df89c-124">A confusing concept is that while a send port in an orchestration is bound to a physical send port, this does not preclude that message from getting delivered to other subscribers.</span></span> <span data-ttu-id="df89c-125">也就是说，如果另一个发送端口恰好有一个订阅，通过其筛选器，发送到绑定端口，这两个发送端口接收消息的消息。</span><span class="sxs-lookup"><span data-stu-id="df89c-125">That is, if another send port happens to have a subscription, through its filters, for the message being sent to the bound port, both send ports receive the message.</span></span> <span data-ttu-id="df89c-126">绑定仅创建订阅，以便从业务流程始终发送的消息与绑定的发送端口的条件相匹配。</span><span class="sxs-lookup"><span data-stu-id="df89c-126">Binding simply creates the subscription such that the message sent from the orchestration always matches the criteria for the bound send port.</span></span> <span data-ttu-id="df89c-127">同样，业务流程端口绑定到接收端口创建相应的订阅根据消息类型和接收端口 id。</span><span class="sxs-lookup"><span data-stu-id="df89c-127">Likewise, the orchestration port bound to a receive port creates the appropriate subscription based on message type and receive port ID.</span></span> <span data-ttu-id="df89c-128">订阅确保传入和传出业务流程的消息传送给绑定的端口，但消息仍通过相同的发布和订阅机制前面所述。</span><span class="sxs-lookup"><span data-stu-id="df89c-128">The subscriptions guarantee that the messages going in and out of the orchestration get delivered to the bound ports, but the messages still go through the same publish and subscribe mechanism described earlier.</span></span>  
  
 <span data-ttu-id="df89c-129">可能最常被误解、 误用和未充分使用的绑定选项是直接绑定选项。</span><span class="sxs-lookup"><span data-stu-id="df89c-129">Probably the most misunderstood, and misused or underused binding option is the Direct binding option.</span></span> <span data-ttu-id="df89c-130">直接绑定允许业务流程将消息发布到 MessageBox 数据库使用不同的路由属性很多像发布消息由接收位置。</span><span class="sxs-lookup"><span data-stu-id="df89c-130">Direct binding allows an orchestration to publish messages to the MessageBox database with varying routing properties much like messages are published by receive locations.</span></span> <span data-ttu-id="df89c-131">在简单的直接消息传送，消息发布到 MessageBox 使用升级属性，如接收到 BizTalk Server 的任何其他已发布的消息路由。</span><span class="sxs-lookup"><span data-stu-id="df89c-131">In simple direct messaging, the message is published to the MessageBox with its promoted properties to be routed like any other published message received into BizTalk Server.</span></span> <span data-ttu-id="df89c-132">这使任何订户都可以接收此消息，但要求至少一个订阅服务器上存在，或业务流程将收到路由故障错误。</span><span class="sxs-lookup"><span data-stu-id="df89c-132">This enables any subscriber to receive this message, but requires that at least one subscriber exist or the orchestration will receive a routing failure error.</span></span>

 <span data-ttu-id="df89c-139">最后一个选项是使用合作伙伴业务流程中，在调用业务流程和调用的业务流程中的端口配置使用的相同共享端口类型和在端口配置中，选择相同的端口。</span><span class="sxs-lookup"><span data-stu-id="df89c-139">The final option is to use a partner orchestration in which, in both the calling orchestration and the called orchestration, the port is configured using the same shared port type and in the port configuration, the same port is selected.</span></span> <span data-ttu-id="df89c-140">例如，在 Orch1 和 Orch2 中都选定 Orch2.MyDirectPort。</span><span class="sxs-lookup"><span data-stu-id="df89c-140">For example, in both Orch1 and Orch2, Orch2.MyDirectPort is selected.</span></span> <span data-ttu-id="df89c-141">此类型的绑定设置了一种订阅，接收业务流程基于发送业务流程类型、 端口名称和操作名称。</span><span class="sxs-lookup"><span data-stu-id="df89c-141">This type of binding sets up a subscription for the receiving orchestration based on the sending orchestration type, the port name, and the operation name.</span></span> <span data-ttu-id="df89c-142">这样再次确保消息获取路由到正确的实例。</span><span class="sxs-lookup"><span data-stu-id="df89c-142">This again ensures that the messages get routed to the correct instance.</span></span>  
  
 <span data-ttu-id="df89c-143">所有直接消息传送选项使用基本的发布和订阅模型。</span><span class="sxs-lookup"><span data-stu-id="df89c-143">All of the direct messaging options use the underlying publish and subscribe model.</span></span> <span data-ttu-id="df89c-144">这些选项之间的区别是用于创建订阅和路由，在属性中，在使用情况下它们有助于解决。</span><span class="sxs-lookup"><span data-stu-id="df89c-144">The difference between these options is in the properties that are used for creating subscriptions and routing, and in the use cases they help solve.</span></span>  
  
 <span data-ttu-id="df89c-145">使用在业务流程直接绑定的端口时遇到的一个常见问题是业务流程可能发布被订阅到一条消息。</span><span class="sxs-lookup"><span data-stu-id="df89c-145">One common problem encountered when using direct bound ports in orchestrations is that an orchestration may publish a message that it is also subscribed to.</span></span> <span data-ttu-id="df89c-146">例如，业务流程配置为由 PurchaseOrder 消息激活。</span><span class="sxs-lookup"><span data-stu-id="df89c-146">For example, an orchestration is configured to be activated by a PurchaseOrder message.</span></span> <span data-ttu-id="df89c-147">此业务流程使用直接端口将 PurchaseOrder 消息发布到 MessageBox。</span><span class="sxs-lookup"><span data-stu-id="df89c-147">This orchestration uses a direct port to publish the PurchaseOrder message to the MessageBox.</span></span> <span data-ttu-id="df89c-148">但是，除了按预期方式接收消息，因为它也具有 PurchaseOrder 消息的订阅启动的业务流程的另一个实例。</span><span class="sxs-lookup"><span data-stu-id="df89c-148">However, in addition to receiving the message as expected, another instance of an orchestration is started because it too had a subscription for PurchaseOrder messages.</span></span> <span data-ttu-id="df89c-149">该处理进入无穷循环，并可能需要一些时间让开发人员可以找出发生了什么。</span><span class="sxs-lookup"><span data-stu-id="df89c-149">The processing gets into an endless loop and it may take some time for a developer to figure out what has happened.</span></span>  
  
## <a name="correlation"></a><span data-ttu-id="df89c-150">Correlation</span><span class="sxs-lookup"><span data-stu-id="df89c-150">Correlation</span></span>  
 <span data-ttu-id="df89c-151">*相关*业务流程中是用于相关的消息接收到同一个正在运行的业务流程实例的机制。</span><span class="sxs-lookup"><span data-stu-id="df89c-151">*Correlation* in orchestrations is the mechanism for receiving related messages into the same running orchestration instance.</span></span> <span data-ttu-id="df89c-152">在业务流程设计器中，开发人员按照以下常规步骤，使用相关：</span><span class="sxs-lookup"><span data-stu-id="df89c-152">In the Orchestration Designer a developer follows these general steps to use a correlation:</span></span>  
  
- <span data-ttu-id="df89c-153">定义包含用于消息相关的升级的属性的相关类型。</span><span class="sxs-lookup"><span data-stu-id="df89c-153">Defines a correlation type that includes the promoted properties that are used to relate messages.</span></span>  
  
- <span data-ttu-id="df89c-154">定义一个相关集，后者定义只是相关类型的实例。</span><span class="sxs-lookup"><span data-stu-id="df89c-154">Defines a correlation set that is an instance of the correlation type just defined.</span></span>  
  
- <span data-ttu-id="df89c-155">用于发送和接收端口，指定是否初始化或沿用给定的相关集。</span><span class="sxs-lookup"><span data-stu-id="df89c-155">For the send and receive ports, specifies whether they initiate or follow a given correlation set.</span></span>  
  
  <span data-ttu-id="df89c-156">实例订阅时需考虑启动相关集，因为这是为所有沿用此相关集接收消息的这些端口创建订阅时。</span><span class="sxs-lookup"><span data-stu-id="df89c-156">Instance subscriptions come into play when a correlation set is initiated, as this is when subscriptions are created for all of those ports that follow this correlation set to receive messages.</span></span> <span data-ttu-id="df89c-157">相关类型定义了要用于相关的属性，因为业务流程引擎可以从正在发送或接收初始化操作的消息中提取这些属性。</span><span class="sxs-lookup"><span data-stu-id="df89c-157">Because the correlation type defines the properties to be used for correlation, the orchestration engine can extract these properties from the message being sent or received by the initiating action.</span></span> <span data-ttu-id="df89c-158">然后使用这些值来定义订阅的所有剩余操作的沿用此相关集。</span><span class="sxs-lookup"><span data-stu-id="df89c-158">These values are then used to define subscriptions for all of the remaining actions which follow this correlation set.</span></span>  
  
  <span data-ttu-id="df89c-159">接收到 BizTalk Server 和用于相关的消息具有正确定义和升级到消息上下文及其已提升的属性至关重要。</span><span class="sxs-lookup"><span data-stu-id="df89c-159">It is important that messages received into BizTalk Server and intended for use in a correlation have their promoted properties correctly defined and promoted to the message context.</span></span> <span data-ttu-id="df89c-160">大多数属性获取升级时最初收到消息时，在管道的拆装器组件提取值。</span><span class="sxs-lookup"><span data-stu-id="df89c-160">Most properties get promoted when a disassembler component in a pipeline extracts the values when the message is initially received.</span></span> <span data-ttu-id="df89c-161">出于此原因，不能使用直通接收管道来接收必须与正在运行的业务流程实例相关的消息。</span><span class="sxs-lookup"><span data-stu-id="df89c-161">For this reason, it is not possible to use the PassThrough receive pipeline to receive messages that must be correlated to a running instance of an orchestration.</span></span> <span data-ttu-id="df89c-162">当您使用 SOAP 接收适配器接收相关的消息，因为直通管道是接收管道的默认值时使用 Web Services 发布向导时，会出现此问题。</span><span class="sxs-lookup"><span data-stu-id="df89c-162">This issue arises when you use the SOAP receive adapter to receive correlated messages, because the PassThrough pipeline is the default value for the receive pipeline when using the Web Services Publishing Wizard.</span></span>
