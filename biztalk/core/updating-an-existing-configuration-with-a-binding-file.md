# <a name="updating-an-existing-configuration-with-a-binding-file"></a>使用绑定文件更新现有配置
绑定文件中的信息将取代现有的配置信息。 如果绑定文件中项目的名称与现有配置中的项目的名称相匹配，绑定文件中的项目的项目将更新现有配置中导入绑定文件时。  
  
 使用绑定文件项目更新现有项目时, 应遵循某些规则。 本主题讨论使用绑定文件中的项目更新现有配置中的项目时应遵循的规则。  
  
 本部分假设时该文件导入，并且不讨论其中的绑定文件包含无效值的任何方案存在绑定文件中有效的值。  
  
## <a name="rules-followed-by-biztalk-server-when-updating-a-configuration-with-a-binding-file"></a>使用绑定文件更新配置时遵循的 BizTalk Server 规则  
 绑定文件中的匹配项目更新现有项目时，BizTalk Server 遵循某些规则。 一般情况下，将应用以下规则：  
  
- 文本框和 BizTalk Server 用户界面 （如 BizTalk Server 管理控制台或 BizTalk 浏览器），通过配置项目时显示的复选框必须设置为特定值或为空。 绑定文件中的项目提供的值将相应地设置已更新项的用户界面值。  
  
- 通过 BizTalk Server 用户界面配置项目必须设置为特定值或设置为"None"时显示的下拉列表框。 绑定文件中的项目提供的值将相应地设置已更新项的用户界面值。  
  
- 从绑定文件中的相应项的列表更新时通过 BizTalk Server 用户界面配置项目显示的数据网格视图。 除非数据网格视图列表绑定到端口或接收位置，绑定文件中的列表始终会覆盖与数据网格视图关联的列表。 在这种情况下与现有的数据网格视图列表合并绑定文件中的列表。  
  
- 绑定文件中的项目由主键值标识。 项目永远不会设置为 null，绑定文件中的所有项目中的用户界面，因此与主键相关联的值必须设置其主键值。 如果绑定文件中项目的主键与关联的值与现有配置项目的主键与关联的值相匹配然后认为这些项目可以相同或相匹配。 如果绑定文件项目与现有项目相同然后更新现有项目，与绑定文件项目下表中所述。 如果绑定文件中的项目包含唯一的主键值然后新项目中创建 BizTalk Server 配置导入绑定文件时。

## <a name="unbinding-behavior-when-updating-existing-artifacts-with-matching-artifacts-in-a-binding-file"></a>正在取消绑定行为的绑定文件中的匹配项目更新现有项目时  
 绑定文件项目通常配置为引用其他项目，例如接收端口通常配置为引用接收位置。 在此方案中，接收端口是父项目和接收位置是子项目。 接收端口是**显式**配置为引用接收位置和接收位置然后**隐式**引用接收端口。 如果未完全配置父项目绑定中的文件存在，例如接收端口的接收位置，不会对其进行配置，则它们将未完全配置而不考虑其状态中的现有导入绑定文件后配置。 例如，如果你有现有的接收端口 myRP 配置了接收位置 myRL 的相同接收端口 myRP 绑定文件中的是**不**配置接收位置 myRL 之后将绑定文件项,优先顺序。 此示例接收端口 myRP 将不会配置与接收位置导入绑定文件，因此您可以有效后**未绑定**地从 myRP。  
  
 此规则仅应用于导入进行显式引用和导入具有隐式引用的项目时不。 因此，如果您导入了隐式引用的映射 （显式由引用） 10 个其他项目您无需担心该映射会从隐式引用的项目取消绑定。
